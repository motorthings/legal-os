Governance Contract Pattern

Every function exposes three endpoints. The governance layer polls them independently. It does not know what a "contract" is or what a "matter" is. It knows: is this function healthy? Is it performing? Is it compliant?

Endpoint 1
Health
GET /api/{function}/health
{"status": "healthy", "uptime": 99.97, "version": "2.1.0", "last_error": null}
If a function is down, the operations dashboard shows it before anyone complains. Uptime and deployment history tracked per function.
Endpoint 2
Metrics
GET /api/{function}/metrics
{"adoption_rate": 0.72, "processing_volume": 1847, "avg_response_time_ms": 8200, "accuracy": 0.94, "flag_rate": 0.12, "cost_per_operation": 0.043}
Every function quantifies its own value. Adoption, speed, accuracy, and cost — measured per invocation, aggregated per period.
Endpoint 3
Targets
GET /api/{function}/targets
{"standards": ["accuracy>0.85","bias<0.05"], "current_scores": {"accuracy":0.94,"bias":0.02}, "drift_status": "stable", "last_evaluated": "2026-07-07T09:00:00Z"}
Model drift detection. Evaluation standards versioned. Any function falling below threshold triggers automatic review. The system polices itself.

Infrastructure
Enterprise LLM Architecture

The platform assumes an enterprise-grade LLM deployment — private instance or enterprise API tier with contractual data protection.

Principle 1
Data Privacy
Enterprise agreements cover all API calls. No training on client data. No retention. No leakage between clients. All data stays within the contractual trust boundary. Embedding and analysis run inside the firm's enterprise infrastructure.
Principle 2
Single Provider Interface
The LLM Gateway auto-detects provider from configuration. Anthropic enterprise endpoint, Azure OpenAI, or AWS Bedrock — functions call the same interface. Provider changes are configuration changes, not code changes.
Principle 3
Usage Tracking
Every LLM call is metered: tokens in, tokens out, model version, latency, cost. This feeds directly into the Metrics Pipeline for per-client cost attribution. The question "how much did we spend on AI this month?" is answered in the dashboard.
Security
Data Isolation Architecture

Row-Level Security is the enforcement mechanism. Not application logic. Not middleware. PostgreSQL RLS policies that cannot be bypassed — even with a direct database connection.

Boundary 1
Client Isolation
Every row in every table has a client_id column. PostgreSQL RLS enforces: client_id = current_user_client_id(). Client A's data cannot touch Client B's queries — same database, same tables, structural guarantee.
Boundary 2
Practice Group Scoping
Additional RLS policies scope by practice group. IP sees only IP matters. Litigation sees only litigation matters. Employment legal walled from commercial legal at the database layer. Not application filtering. Not configurable per-request. Structural.
Boundary 3
Immutable Audit Trail
Audit records are append-only. Write once, read by authorized roles. No update. No delete. Every evaluation, every override, every model call — permanently recorded. When regulators or clients audit, the trail is already there.
